# Workflow: Outcome-Oriented Proposal

Создание коммерческого предложения, которое продаёт **трансформацию бизнеса**, а не объём работ.

> **Принцип:** Клиент покупает не код/дизайн/бота — клиент покупает решение своей бизнес-проблемы.

## Предварительные условия

- Проведена Discovery-встреча (см. `/discovery`)
- Заполнен Discovery Summary документ
- Определены Problem Statement, Success Metrics и Value of Outcome

## Структура предложения

### 1. Контекст и понимание бизнеса

Покажите клиенту, что вы **поняли** его бизнес:
- Краткое описание бизнес-модели клиента (своими словами)
- Главная проблема / возможность, которую вы обнаружили
- Стоимость этой проблемы для бизнеса

> Это демонстрирует вашу экспертизу и отличает вас от команд, которые просто «делают по ТЗ».

### 2. Предлагаемое решение (Outcome Focus)

Описание того, **что изменится** в бизнесе клиента:
- Как решение трансформирует текущую ситуацию
- Какие конкретные бизнес-результаты ожидаются
- Привязка к метрикам успеха из Discovery

**Формат:** «В результате проекта ваш бизнес получит X, что приведёт к Y»

НЕ: «Мы сделаем 10 экранов, REST API и админ-панель»

### 3. Подход и этапы

Краткое описание процесса (не ТЗ, а методология):
- Фазы проекта и что клиент получает на каждой
- Ключевые точки валидации с бизнесом
- Как обеспечивается качество outcome

### 4. Ценообразование (Value-Based)

Формула обоснования цены:

```
Стоимость проблемы для клиента: $X / месяц
Ожидаемый ROI от решения: Y% за Z месяцев
Инвестиция в проект: $N (= X * коэффициент)
```

**Правила:**
- Цена должна составлять 5–20% от годовой ценности outcome для клиента
- Никогда не привязывать цену к часам работы
- Если клиент зарабатывает $X/день, проект может стоить несколько дней его дохода

### 5. Почему мы

- Релевантные кейсы (см. `/case-study`)
- Уникальная экспертиза (Capability)
- Что отличает от команд, которые «работают по ТЗ»

### 6. Следующие шаги

Чёткий call-to-action:
- Что нужно от клиента для старта
- Временные рамки
- Контактная информация

## Шаблон КП

```markdown
# [Название проекта] — Предложение

## Мы поняли вашу задачу

[2-3 абзаца: бизнес-модель клиента, главная проблема, стоимость проблемы]

## Что вы получите

[Описание outcome — бизнес-результат, НЕ список фич]

### Ожидаемые результаты
- Метрика 1: текущее значение → целевое значение
- Метрика 2: текущее значение → целевое значение
- Метрика 3: текущее значение → целевое значение

## Наш подход

### Фаза 1: [Название] (X недель)
[Что происходит и что клиент получает]

### Фаза 2: [Название] (X недель)
[Что происходит и что клиент получает]

### Фаза 3: [Название] (X недель)
[Что происходит и что клиент получает]

## Инвестиция

|                        | Сумма     |
| ---------------------- | --------- |
| Проект                 | $X        |
| Ожидаемый ROI (12 мес) | $Y        |
| Окупаемость            | Z месяцев |

## Почему Ivannikov Pro

- [Релевантный кейс 1]
- [Релевантный кейс 2]
- [Уникальная экспертиза]

## Следующие шаги

1. Подтверждение предложения
2. Подписание договора и оплата первого этапа
3. Kick-off встреча — [дата]
```

## Чек-лист перед отправкой

- [ ] КП начинается с проблемы бизнеса, а не с описания услуг
- [ ] Ни одного упоминания технологий без привязки к бизнес-ценности
- [ ] Цена обоснована через ROI / ценность для бизнеса
- [ ] Есть конкретные метрики успеха
- [ ] Клиент увидит себя и свой бизнес в документе
- [ ] Есть чёткий next step

## Anti-паттерны

- ❌ Список фич вместо описания outcome
- ❌ Почасовая оценка
- ❌ «Мы используем React, Node.js, PostgreSQL...» без контекста ценности
- ❌ Длинное КП без executive summary
- ❌ Отсутствие привязки к бизнес-метрикам
