# Workflow: Project Brief (Outcome-First)

Создание брифа проекта, который начинается с **бизнес-целей**, а не с технического задания.

> **Принцип:** Бриф отвечает на вопрос «Какую проблему мы решаем?», а не «Что нужно сделать?»

## Когда использовать

- После завершения Discovery (см. `/discovery`)
- Перед началом проектирования и разработки
- Как внутренний документ для команды

## Структура брифа

### 1. Business Context

```markdown
## Бизнес-контекст

**Клиент:** [Название / описание без NDA-нарушений]
**Ниша:** [Сфера деятельности]
**Масштаб:** [Оборот / команда / количество клиентов]
**Бизнес-модель:** [Как зарабатывает]
```

### 2. Problem Statement

Одно предложение, определяющее главную проблему:

```markdown
## Проблема

[Кто] не может [что], потому что [почему], что приводит к [последствие для бизнеса].

**Стоимость проблемы:** [$X / месяц] или [Y часов потерянного времени]
```

### 3. Desired Outcome

```markdown
## Желаемый результат (Outcome)

В результате проекта [клиент] сможет [что], что приведёт к [бизнес-результат].

### Метрики успеха
1. [Метрика 1]: текущее значение → целевое значение
2. [Метрика 2]: текущее значение → целевое значение
3. [Метрика 3]: текущее значение → целевое значение

### Временные рамки
- MVP / первая ценность: [срок]
- Полная реализация: [срок]
```

### 4. Solution Hypothesis

```markdown
## Гипотеза решения

Мы считаем, что [тип решения] поможет [клиенту] достичь [outcome], потому что [обоснование].

### Ключевые компоненты
1. [Компонент 1] — решает [какую часть проблемы]
2. [Компонент 2] — решает [какую часть проблемы]
3. [Компонент 3] — решает [какую часть проблемы]
```

### 5. Constraints & Risks

```markdown
## Ограничения и риски

### Ограничения
- Бюджет: ...
- Сроки: ...
- Технические: ...
- Юридические / регуляторные: ...

### Риски
| Риск | Вероятность            | Влияние | Митигация |
| ---- | ---------------------- | ------- | --------- |
| ...  | Высокая/Средняя/Низкая | ...     | ...       |
```

### 6. Technical Approach (краткий)

```markdown
## Технический подход

**Важно:** Технологии выбираются как инструмент достижения outcome, а не ради самих себя.

- **Платформа:** [почему именно эта — привязка к outcome]
- **Архитектура:** [почему — привязка к масштабированию/скорости]
- **Интеграции:** [с чем и зачем]
```

### 7. Success Validation

```markdown
## Валидация успеха

### Как мы узнаем, что проект успешен?
1. [Метрика 1] достигнута в течение [срок]
2. [Метрика 2] достигнута в течение [срок]
3. Клиент подтверждает, что [качественный критерий]

### Точки проверки
- После Фазы 1: проверяем [что]
- После Фазы 2: проверяем [что]
- Через 30 дней после запуска: проверяем [что]
```

## Чек-лист брифа

- [ ] Начинается с бизнес-контекста, а не с описания фич
- [ ] Problem Statement — одно чёткое предложение
- [ ] Есть измеримые метрики успеха
- [ ] Технический подход обоснован через бизнес-ценность
- [ ] Определены риски и ограничения
- [ ] Есть план валидации outcome
- [ ] Документ понятен нетехническому стейкхолдеру
